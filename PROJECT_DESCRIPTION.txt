1. SYSTEM OVERVIEW
================================================================================

The Exam Proctoring System is a web-based application designed to conduct 
online examinations with real-time proctoring capabilities. The system ensures 
academic integrity by monitoring students during exams using computer vision 
and audio analysis.

KEY FEATURES:
- User Registration and Authentication (JWT-based)
- Face Recognition for Identity Verification
- Real-time Video Proctoring
- Audio Monitoring (Voice Detection)
- Head Pose Detection
- Multiple Person Detection
- Exam Management (30+ domain-specific exams)
- Question Bank Management
- Session Management
- Violation Logging
- Automated Report Generation

================================================================================
2. SYSTEM ARCHITECTURE

4. DATABASE SCHEMA (SQLite)
================================================================================

4.1 USERS TABLE
   - id (INTEGER PRIMARY KEY)
   - user_id (TEXT UNIQUE)
   - full_name (TEXT)
   - student_id (TEXT)
   - email (TEXT UNIQUE)
   - phone (TEXT)
   - course (TEXT)
   - role (TEXT) - 'student', 'candidate', 'staff'
   - password_hash (TEXT)
   - photo_path (TEXT)
   - encoding_path (TEXT) - Face encoding file path
   - notes (TEXT)
   - created_at (TEXT)

4.2 EXAMS TABLE
   - exam_id (INTEGER PRIMARY KEY)
   - exam_title (TEXT)
   - description (TEXT)
   - domain (TEXT) - Programming, Data Science, Web Development, etc.
   - difficulty (TEXT) - Beginner, Intermediate, Advanced
   - duration (INTEGER) - in minutes
   - total_questions (INTEGER)
   - is_active (INTEGER) - 0 or 1
   - created_at (TEXT)

4.3 EXAM_QUESTIONS TABLE
   - question_id (INTEGER PRIMARY KEY)
   - exam_id (INTEGER) - Foreign Key to exams
   - question (TEXT)
   - options (TEXT) - JSON array of options
   - correct_answer (INTEGER) - Index of correct option
   - question_order (INTEGER)

4.4 SESSIONS TABLE
   - session_id (INTEGER PRIMARY KEY)
   - user_id (TEXT) - Foreign Key to users
   - exam_id (INTEGER) - Foreign Key to exams
   - start_time (TEXT)
   - end_time (TEXT)
   - status (TEXT) - 'active' or 'completed'
   - created_at (TEXT)

4.5 REPORTS TABLE
   - report_id (INTEGER PRIMARY KEY)
   - session_id (INTEGER) - Foreign Key to sessions
   - user_id (TEXT) - Foreign Key to users
   - exam_id (INTEGER) - Foreign Key to exams
   - total_questions (INTEGER)
   - correct_answers (INTEGER)

   - percentage (REAL)
   - submitted_at (TEXT)

4.6 VIOLATION_LOGS TABLE
   - violation_id (INTEGER PRIMARY KEY)
   - session_id (INTEGER) - Foreign Key to sessions
   - violation_type (TEXT) - NO_FACE, MULTIPLE_FACES, HEAD_POSE, VOICE_START, VOICE_VIOLATION, etc.
   - violation_details (TEXT)
   - timestamp (TEXT)
   - severity (TEXT) - 'low', 'medium', 'high'

6. USE CASES
================================================================================

UC1: REGISTER USER
   Actor: Student/Candidate
   Precondition: User not registered
   Main Flow:
   1. User accesses registration page
   2. User fills registration form (name, email, password, etc.)
   3. User captures/uploads photo
   4. System validates form data
   5. System detects face in photo
   6. System computes face encoding
   7. System stores user data and encoding
   8. System returns success message
   Postcondition: User account created

UC2: LOGIN
   Actor: Student/Candidate/Staff
   Precondition: User registered
   Main Flow:
   1. User enters email and password
   2. System validates credentials
   3. System generates JWT token
   4. System returns token to client
   5. Client stores token
   Postcondition: User authenticated

UC3: VERIFY IDENTITY
   Actor: Student/Candidate
   Precondition: User logged in
   Main Flow:
   1. User accesses verification page
   2. System requests camera access
   3. User positions face in camera
   4. User captures photo
   5. System extracts face encoding
   6. System compares with registered encoding
   7. System returns match result
   Postcondition: Identity verified or rejected

UC4: VIEW AVAILABLE EXAMS
   Actor: Student/Candidate
   Precondition: User logged in and verified
   Main Flow:
   1. User accesses home page
   2. System fetches available exams from database
   3. System displays exams with filters
   4. User can filter by domain
   Postcondition: Exams displayed

UC5: START EXAM SESSION
   Actor: Student/Candidate
   Precondition: User verified, exam selected
   Main Flow:
   1. User selects exam
   2. System creates session record
   3. System records start time
   4. System loads exam questions
   5. System initializes proctoring
   Postcondition: Exam session active

UC6: TAKE EXAM WITH PROCTORING
   Actor: Student/Candidate
   Precondition: Exam session started
   Main Flow:
   1. System displays questions
   2. System starts video stream
   3. System captures frames periodically
   4. System analyzes frames for violations
   5. System monitors audio
   6. User answers questions
   7. System logs violations if detected
   8. System displays warnings
   Postcondition: Exam in progress

UC7: DETECT PROCTORING VIOLATIONS
   Actor: System
   Precondition: Exam session active
   Main Flow:
   1. System captures video frame
   2. System detects faces
   3. System estimates head pose
   4. System checks for multiple persons
   5. System monitors audio levels
   6. If violation detected, system logs it
   7. System displays warning to user
   Postcondition: Violation logged

UC8: SUBMIT EXAM
   Actor: Student/Candidate
   Precondition: Exam in progress
   Main Flow:
   1. User clicks submit button
   2. System calculates score
   3. System ends session
   4. System creates report
   5. System displays results
   Postcondition: Exam completed, report generated

UC9: VIEW REPORT
   Actor: Student/Candidate/Staff
   Precondition: Exam completed
   Main Flow:
   1. User requests report
   2. System fetches report data
   3. System fetches violation logs
   4. System displays score and violations
   5. User can download report
   Postcondition: Report displayed

UC10: MANAGE EXAMS (Future)
   Actor: Staff/Admin
   Precondition: Staff logged in
   Main Flow:
   1. Staff accesses exam management
   2. Staff creates/edits exams
   3. Staff adds questions
   4. System stores exam data
   Postcondition: Exam data updated

10. PROCTORING FEATURES DETAIL
================================================================================

10.1 FACE DETECTION
    - Primary: YuNet ONNX Model (High accuracy)
    - Fallback 1: Caffe DNN (Medium accuracy)
    - Fallback 2: Haar Cascade (Basic detection)
    - Detects face count, position, landmarks
    - Minimum face size: 80px (configurable)

10.2 HEAD POSE DETECTION
    - Uses facial landmarks (eyes, nose)
    - Calculates head rotation (Left/Right/Center)
    - Threshold: 0.25 (configurable)
    - Alert duration: 3.0 seconds (configurable)
    - Severity levels based on deviation

10.3 MULTIPLE PERSON DETECTION
    - Counts faces in frame
    - Threshold: 1 person allowed
    - Logs violation if > 1 face detected
    - Severity: High

10.4 VOICE/AUDIO MONITORING
    - RMS (Root Mean Square) level detection
    - Threshold: 0.08 (normalized, configurable)
    - Alert duration: 2.0 seconds (configurable)
    - Tracks voice start/stop events
    - Periodic monitoring

10.5 VIOLATION TYPES
    - NO_FACE: No face detected in frame (High severity)
    - MULTIPLE_FACES: Multiple persons detected (High severity)
    - HEAD_POSE: Head turned left/right (Medium severity)
    - VOICE_START: Voice detected (Low severity)
    - VOICE_VIOLATION: Sustained voice detected (Medium severity)
    - VOICE_DETECTED: Voice over threshold (Low severity)

10.6 PROCTORING PARAMETERS (Configurable)
    - head_pose_threshold: 0.25
    - head_pose_alert_duration: 3.0 seconds
    - voice_threshold: 0.08
    - voice_alert_duration: 2.0 seconds
    - multiple_persons_threshold: 1
    - min_face_size: 80 pixels
    - confidence_threshold: 0.6

18.1 EXAM TAKING ACTIVITY

    Start → Receive Frame → Try YuNet → Success? → Yes → Return Faces
    → No → Try Caffe → Success? → Yes → Return Faces
    → No → Try Haar → Return Faces → End

18.3 VIOLATION DETECTION ACTIVITY
    Start → Detect Faces → Count Faces → = 0? → Log NO_FACE
    → > 1? → Log MULTIPLE_FACES
    → = 1? → Estimate Head Pose → Check Threshold → 
    Violation? → Log HEAD_POSE → Check Audio → Over Threshold? → 
    Log VOICE_VIOLATION → End
